Section 1: Platform Architecture (15 min)

Anypoint Platform is MuleSoft's unified integration platform. It covers the entire API lifecycle — from designing a spec to deploying a production app to securing it with policies. Before you build anything, you need to know where everything lives. Think of this as learning the kitchen before you start cooking.

The platform has five major components that you'll use throughout this course. Each one handles a different phase of the lifecycle, and they're connected by a shared asset catalog called Exchange.

The flow is straightforward: you design an API specification in Design Center, publish it to Exchange (the shared asset catalog), scaffold a project from that spec in Anypoint Code Builder, implement your business logic, deploy to CloudHub via Runtime Manager, and then apply security policies in API Manager. Every tool hands off to the next.

Code Snippets: What Each Component Produces

Snippet 1: What Design Center Produces

A RAML 1.0 specification that defines your API contract. This is what you'll create in Module 3:

#%RAML 1.0
title: Customer System API
version: v1
baseUri: http://localhost:8081/api
/customers:
  get:
    description: List all customers

Snippet 2: What ACB Produces

When you scaffold a project from a RAML spec, ACB generates Mule XML flows. This is the skeleton you'll flesh out in Module 4:

<flow name="customer-flow">
  <http:listener config-ref="HTTP_Listener" path="/api/customers" method="GET"/>
  <logger level="INFO" message="Request received for /api/customers"/>
</flow>

Snippet 3: MCP Server Configuration

This JSON config connects your AI IDE to Anypoint Platform via the MCP server:

{
  "mcpServers": {
    "mulesoft": {
      "command": "npx",
      "args": ["-y", "@mulesoft/mcp-server", "start"],
      "env": {
        "ANYPOINT_CLIENT_ID": "<YOUR_CLIENT_ID>",
        "ANYPOINT_CLIENT_SECRET": "<YOUR_CLIENT_SECRET>",
        "ANYPOINT_REGION": "PROD_US"
      }
    }
  }
}
Connected Apps

The CLIENT_ID and CLIENT_SECRET come from a Connected App configured in Access Management. You'll create one in Module 6 when you set up Salesforce integration. For the MCP server, use an "App acts on its own behalf" type with Client Credentials grant.

Snippet 4: What a Deployed Endpoint Looks Like

By Module 7, your API will be live on CloudHub with a public URL:

curl https://customer-enrichment-api.us-e1.cloudhub.io/api/customers/1
{
  "id": "ACCT-001",
  "name": "Leanne Graham",
  "company": "Romaguera-Crona",
  "score": 85,
  "segment": "premium"
}

That enriched response combines data from Salesforce, a REST enrichment API, a PostgreSQL database, and a SOAP address validation service — all orchestrated by a single Mule flow.

Section 2: Component Deep Dive (10 min)

Now let's look at each component in more detail. For each one, you'll learn what it does, how it fits into the Customer 360 project, and which module you'll use it in.

Design Center

[SCREENSHOT: Design Center — API Designer with RAML spec]

Where you'll design the Customer System API spec in Module 3. Design Center supports RAML 1.0 and OpenAPI 3.0 with both a visual editor and a code editor. It includes a built-in mocking service so you can test your API endpoints before writing any implementation code.

For the Customer 360 project, you'll define resources like /customers and /customers/{customerId}, data types for Customer and Contact, and example responses — all before writing a single line of Mule XML.

Used in: Module 3

Instructor Note

Create a blank spec, show both visual and code editors. Toggle the mocking service on — send a GET request to show it returns example data. Delete the spec when done.

Exchange

[SCREENSHOT: Exchange — Salesforce Connector page]

The marketplace where you'll find connectors (Salesforce, HTTP, Database), templates, examples, and your own published APIs. Think of it as npm or Maven for MuleSoft assets. When you publish your RAML spec in Module 3, it lands here. When you scaffold a project from that spec in Module 4, you pull from here.

For the Customer 360 project, you'll consume the Salesforce Connector, HTTP Request Connector, Database Connector, and Web Service Consumer (SOAP) — all found in Exchange.

Used in: Modules 3, 4, 6

Instructor Note

Search "Salesforce" — show the connector page, version, dependency snippet. Search "HTTP" — show it. Search "Database" — show it. Say: "These three connectors are what you'll use in Module 4." Show a Salesforce migration template.

Anypoint Code Builder (ACB)

[SCREENSHOT: ACB — Canvas view with flow components]

Where you'll build all your Mule flows from Module 4 onward. ACB is a VS Code extension — it works in VS Code, Cursor, or as a cloud IDE in the browser. It has a visual canvas for drag-and-drop flow building and an XML editor. Both views stay in sync: edit in one, see the change in the other.

ACB is MuleSoft's strategic IDE direction. All new AI-assisted features (flow generation, spec scaffolding) are ACB-exclusive. Anypoint Studio still works but ACB is where the innovation is happening.

Used in: Modules 4, 5, 6, 7

Instructor Note

Open ACB (cloud IDE or local VS Code with extension). Show the MuleSoft panel. Create a new Mule project (just the wizard, don't finish). Show the canvas view with components, the XML view, and the connector palette on the left.

Runtime Manager

[SCREENSHOT: Runtime Manager — Applications list]

Where your apps run in production. You'll deploy here in Module 7. Runtime Manager supports CloudHub 2.0 (fully managed, container-based), Runtime Fabric (your own Kubernetes), or hybrid (on-prem agents). It shows app status, logs, metrics, and alerts.

CloudHub 2.0 is the current standard — it's container-based and replaces the legacy CloudHub 1.0. For the Customer 360 project, you'll deploy your enrichment API to CloudHub 2.0 and watch it go green.

Used in: Module 7

Instructor Note

Show the applications list (empty now). Click into an environment. Show where logs would appear. Say: "By Day 3 your app will be listed here with a green Started status."

API Manager

[SCREENSHOT: API Manager — Policies configuration]

Where you secure, govern, and monitor your APIs. You'll apply policies here in Module 7 — rate limiting, client ID enforcement, OAuth. API Manager shows analytics: how many calls your API receives, average latency, error rates, and which consumers are calling it.

For the Customer 360 project, you'll create an API instance from your Exchange spec and apply a rate-limiting policy to protect your endpoint from being overwhelmed.

Used in: Module 7

Instructor Note

Show the API instances list. Click where policies get applied. Mention: "We'll come back here on Day 3 to lock down your API."
